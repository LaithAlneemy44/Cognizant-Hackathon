## How It Works Section

### Heading
**A factory floor, not a black box.**

ForgeSpec runs five specialised agents in sequence. Each one has a defined role, a defined input, and a defined output. You can inspect every artifact at every stage.

---

### Agent Cards

**Blueprint**
*Requirements Analysis*
Reads your brief and produces a structured requirements document. Flags ambiguities before they become bugs. Sets the acceptance criteria every downstream stage is measured against.

---

**Atlas**
*Architecture Design*
Designs the technical structure: components, data flows, API contracts, and the rationale for each decision. Every architectural choice traces back to a requirement.

---

**Forge**
*Implementation*
Writes the code against Atlas's architecture and Blueprint's requirements. Not a prototype — functional, readable implementation with a clear spec behind every decision.

---

**Sentinel**
*Testing*
Generates a test suite from the requirements document, then runs it against Forge's output. Tests target specified behaviour, not just implementation details.

---

**Veritas**
*Final Review*
Reviews the complete pipeline output and produces an audit report. Confirms what was delivered, what passed, and what — if anything — was left unaddressed. The gate that most projects skip entirely.

## Traceability Section

### Heading
**Code without context is debt.**

Six months after a release, the hardest question is always: *what was this supposed to do, and why was it built this way?*

ForgeSpec treats traceability as a first-class output, not an afterthought. Every requirement has an ID. Every architectural decision references that ID. Every test maps to a requirement. Veritas checks the full chain before the pipeline closes.

When things break — and they will — you know exactly where to look.

## Architecture Section

### Heading
**Why separate agents?**

A single model asked to go from brief to code in one pass collapses all the work into one opaque step. No checkpoints. No inspectable decisions. No way to know where things went wrong.

ForgeSpec separates the pipeline into distinct disciplines because software development itself is a sequence of distinct disciplines. Requirements analysis is not architecture. Architecture is not implementation. Implementation is not adversarial testing.

Specialised agents mean failures are localised, outputs are inspectable, and every stage can be evaluated on its own terms.
